## 1. Подключение к модели по сети

**Требование:** пользователь подключается к LLM **по сети** — локально (LAN, on-prem) или через интернет. Режим выбирается политикой окружения и ролью, не настройкой «в обход».

| Режим | Пример | Когда |
|-------|--------|-------|
| Локальный | `OLLAMA_BASE_URL=http://192.168.x.x:11434` | production / restricted data |
| Интернет | OpenAI, Anthropic, облачный API | dev, неконфиденциальные задачи, по gating |

**MUST:**

- Единый provider pattern (`tmki_llm/providers.py`, `16_tool_registry.md`).
- Tool gating по `org/role/env` — сеть не открывает доступ к данным выше clearance.
- Аудит каждого сетевого вызова LLM: `trace_id`, провайдер, модель, без PII в payload лога.
- Security-review перед production egress (см. `schemas/security/mvp-security-review.checklist.json`).

**SHOULD:** health-check endpoint провайдера при старте сессии (`check_runtime_health.py` расширить).

## 2. Папка сотрудника на рабочем столе

**Требование:** на рабочем столе Windows у каждого сотрудника — папка с **фамилией** для рабочих файлов.

| Параметр | Значение |
|----------|----------|
| Путь (default) | `%USERPROFILE%\Desktop\{surname}\` |
| Пример | `C:\Users\ivanov\Desktop\Иванов\` |
| Имя папки | Фамилия из `employee.display_name` / HR-реестра (кириллица, как в кадрах) |
| Назначение | Рабочие документы, черновики, выгрузки; не системные файлы |

**MUST:** папка привязана к `employee_id` и `folder_id` в `FolderCatalog` (`storage_backend: filesystem`).

**MUST NOT:** хранить в этой папке HR voice/биометрию (см. §6) — только рабочие файлы.

## 3. Синхронизация с локальным сервером (каждые 5 с)

**Требование:** на локальном сервере хранится полная копия файлов из папки сотрудника на рабочем столе; агент проверяет и дописывает новые/изменённые файлы **каждые 5 секунд**.

```
Desktop\{surname}\  ──watch 5s──►  Local sync server  ──►  ingest / backup / RAG (по policy)
```

| Параметр | Значение |
|----------|----------|
| Интервал опроса | **5 с** (default), настраиваемо `TMKI_DESKTOP_SYNC_INTERVAL_SEC` |
| Направление | Desktop → server (one-way default); конфликты — server wins + audit |
| Триггер ingest | Новый/изменённый файл → `ingest-request` с `source_path`, `folder_id` |
| Схема | `schemas/document/desktop-sync-manifest.schema.json` |

**MUST:** dedup по `content_hash`; аудит `desktop_sync_file_uploaded`.

**SHOULD:** пауза sync при открытом файле (Office lock) — retry на следующем цикле.

## 5. Создание и правка документов

**Требование:**

1. При создании новых документов или правке существующих — **опираться на внутренние нормативные акты, шаблоны и примеры** компании (корпус TMKI / Сатимол).
2. При создании **нового** документа (кроме **договора**) — **не проверять** соответствие российскому законодательству **по умолчанию**.
3. Проверка по внешней нормативной базе (§4) — **только по явному запросу пользователя** или для типа `contract`.

| Тип документа | Проверка по РФ законодательству |
|---------------|--------------------------------|
| Договор (`contract`) | **MUST** (всегда) |
| Прочие новые документы | **MAY** — только по запросу |
| Правка существующих | Шаблоны/внутренние акты; внешняя проверка — по запросу |

Схема политики: `schemas/document/document-creation-policy.schema.json`.

## 6. HR-карточка сотрудника (голос и антропометрия)

**Требование:** в отделе кадров на каждого сотрудника заводится карточка по российскому законодательству **плюс** (при подписанном согласии):

- голосовой профиль (для идентификации, §7);
- антропометрические данные.

| Параметр | Значение |
|----------|----------|
| Согласие | Подписанное соглашение на обработку ПДн — **обязательно** |
| Хранение | Отдельное защищённое хранилище, **не** в RAG/pgvector |
| Доступ | Только HR + Security owner; audit каждого чтения |
| Схема | `schemas/org/employee-hr-card.schema.json` |

**MUST:** соответствие 152-ФЗ; retention policy; право на удаление по запросу сотрудника.

## 7. Распознавание голоса — to-do и планёрки

**Требование:** модель обучена/настроена распознавать голос сотрудников для:

- ведения **to-do листов** (голосовые команды → задачи);
- **планёрок** (транскрипт + action items по спикеру).

**MUST:** идентификация спикера только при наличии согласия (§6) и enrollment голосового профиля.

**SHOULD:** skill «Voice Meeting Assistant» в `13_ai_skills_registry.md`.

Схема сессии: `schemas/voice/voice-session.schema.json`.

## 8. Синтез речи (TTS) и вывод на устройства

**Требование:**

1. Модель отвечает **сгенерированным голосом** (нейро-TTS), без клонирования голоса реального актёра.
2. Модель **выводит информацию** на подключённые устройства: ТВ, планшет, компьютер.

### Рекомендуемый TTS: Piper (бесплатный, офлайн)

| Параметр | Значение |
|----------|----------|
| Движок | [Piper](https://github.com/OHF-Voice/piper1-gpl) (`pip install piper-tts`) |
| Лицензия | GPL-3.0 (бесплатно); голоса — отдельные лицензии на HuggingFace |
| Голоса | 100+ предустановленных, **скачиваются бесплатно** с [rhasspy/piper-voices](https://huggingface.co/rhasspy/piper-voices) |
| RU default | `ru_RU-denis-medium` (`PIPER_VOICE`) |
| Почему Piper | Самый популярный офлайн neural TTS для self-hosted (Home Assistant, Raspberry Pi); real-time на CPU |

Альтернативы (Watchlist): Kokoro (Apache 2.0), MeloTTS (MIT) — при необходимости более «человечного» тембра.

| Компонент | Реализация |
|-----------|------------|
| TTS | `runtime/tmki_voice/tts.py`; `TMKI_TTS_PROVIDER=stub\|piper` |
| Persona | `generated_default` — preset Piper, **не** клон реального человека |
| Cast / display | `tmki_voice/display.py` (backlog) — HDMI, browser cast |

**MUST NOT:** клонировать голос реального актёра без лицензии (отменено требование по голосу Зайцева).

**SHOULD:** fallback — `stub` или системный TTS при недоступности Piper.
